# Sprint 2 - Architecture Decisions

## 1. Real-time via polling, not SSE - for now
The approved V2 architecture recommends SSE for live updates, and the EventBus built in
Sprint 1 was explicitly designed as the seam for it. Sprint 2 still uses polling (React
Query `refetchInterval`, 5-20s depending on the widget) rather than building an SSE
endpoint on top of that bus. Reasoning: Sprint 2's explicit deliverable list asked for "REST
endpoints needed by the UI," this strategy's signal frequency makes a few-second poll
indistinguishable from push for a human trader, and it keeps this sprint's surface area to
REST only, consistent with "do not build the entire platform in one iteration." The EventBus
already exists and is already being consumed (by `SystemStatus`, see below) - wiring an SSE
endpoint to it later is additive, not a rework.

## 2. Trade Timeline is derived, not backed by a new events table
The V2 architecture's long-term design has an append-only `trade_events` table for a true
multi-event timeline. Sprint 2 does not add it - the timeline shown per trade
(`atlas/api/v1/trades.py::build_timeline`) is computed from the single `trades` row's own
fields: entry, latest known price update, exit, AI note. This means it shows the *latest*
price update, not the full history of every `price_update` webhook received. This is stated
plainly in the API response and the frontend UI, not silently simplified. A real multi-event
timeline is a well-scoped, independent follow-up sprint (add the table, backfill nothing,
start appending going forward).

## 3. `SystemStatus`: a second real subscriber on the Sprint 1 EventBus
`atlas/status.py` subscribes to every event type to answer "who have we heard from since
this process started" for the Connection Status panel - deliberately in-memory and reset on
restart (see that module's docstring), distinct from `trades` table history which persists.
This is the first thing other than the logging subscriber to consume the EventBus,
validating that Sprint 1's pub/sub design holds up under a second, differently-shaped
consumer without any change to the publishers in `webhook.py` (one field was added to an
event payload - `error` on `TRADE_AI_ANALYZED` - a payload enrichment, not a new event or a
behavior change).

## 4. `/stats/today` is explicitly a placeholder, computed in Python, not SQL
Aggregates today's trade count/wins/losses/realized PnL by scanning `list_recent(limit=500)`
in Python rather than a SQL `GROUP BY`. Acceptable at this strategy's trade volume (a
handful of trades/day); revisit with real SQL aggregation or the `daily_stats` materialized
table from the V2 architecture when/if that stops being true. "Today" is the UTC calendar
day, not a trading-session boundary - documented in the endpoint's own docstring rather than
silently assumed.

## 5. Frontend stack: Next.js 16 (App Router) + Tailwind v4 + React Query, as originally approved
Node/npm turned out to be available in this environment (just not on PATH by default), so
the approved stack was buildable and testable end-to-end - no fallback to a server-rendered
page was needed. Next.js 16 shipped after this session's training data cutoff for some APIs;
`AGENTS.md` (generated by `create-next-app`) flagged this, and the bundled docs
(`node_modules/next/dist/docs/`) were checked before writing route/params code - the
`params: Promise<{...}>` async-params convention from Next 15 is unchanged in 16.

## 6. A new dev tool: `scripts/dev_seed_server.py`
`atlas.main`'s lifespan refuses to start without a real `DATABASE_URL` (Sprint 1's
deliberate fail-loud behavior). That makes it impossible to run the real backend for
frontend development without a live Postgres. `scripts/dev_seed_server.py` builds a
standalone FastAPI app that reuses the exact same routers but wires `InMemoryTradeRepository`
(the same test double the pytest suite uses) directly into `app.state`, seeded with a few
realistic trades (open/won/lost, including a PickMyTrade failure case). This is a genuine,
documented dev tool - not a placeholder - and is what was used to visually verify this
sprint's UI end-to-end in this session, screenshots included, since no Postgres was
available in this sandbox. Never used in production or in the automated test suite.

## 7. `correlation_id` bug found and fixed during this sprint's verification
While seeding data for the visual check above, `InMemoryTradeRepository.claim_and_forward`
was found to source the stored `correlation_id` from `entry.get("correlation_id")` instead
of the explicit `correlation_id` parameter - a bug present since Sprint 1, masked because
every test payload happened to duplicate `correlation_id` inside the JSON body (matching
real TradingView payloads). `PostgresTradeRepository` never had this bug - it always used the
explicit parameter. Fixed in `atlas/repositories/memory.py`; added
`tests/test_repositories.py` to cover the repository directly (not just through the webhook
route) so a divergence like this can't hide behind a coincidentally-shaped test fixture
again. No production impact: real webhook payloads always include `correlation_id`, and
Postgres was never affected.
